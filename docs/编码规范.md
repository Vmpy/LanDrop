# LanDrop 代码规范 v1.1

## 1. 基本原则

- 所有代码必须清晰、简洁、自解释。
- 优先使用 C++17 标准特性（与 Qt 6.5 兼容）。
- 严格遵守以下格式与命名约定。

***

## 2. 文件组织

### 2.1 单一功能类

- **每个类或功能单元独占一个** **`.cpp`** **源文件和一个对应的** **`.h`** **头文件**。
- 文件名与类名完全一致，大小写敏感：
  - 类 `discovery_service` → `discovery_service.h` / `discovery_service.cpp`
  - 工具函数集合可放在单个文件中，但须有明确的命名空间。

### 2.2 目录结构（随开发可以扩展）

```
src/
├── core/
│   ├── discovery_service.cpp
│   ├── discovery_service.h
│   ├── transfer_server.cpp
│   ├── transfer_server.h
│   ├── file_receiver.cpp
│   ├── file_receiver.h
│   ├── file_sender.cpp
│   ├── file_sender.h
│   ├── protocol.cpp
│   └── protocol.h
├── ui/
│   ├── main_window.cpp
│   ├── main_window.h
│   ├── device_list_widget.cpp
│   ├── device_list_widget.h
│   ├── receive_dialog.cpp
│   └── receive_dialog.h
└── main.cpp
```

- `main.cpp` 仅包含 `main` 函数以及全局初始化代码。

### 2.3 头文件保护

- 使用 `#pragma once` 进行头文件保护（Qt 推荐，所有编译器均支持）。
- 不再使用传统的 `#ifndef` 宏形式。

***

## 3. 编码风格

### 3.1 ALLMAN 风格（括号独立成行）

- **所有大括号（`{`** **和** **`}`）必须独占一行**，包括控制流、函数、类、命名空间等。

示例：

```cpp
void do_something()
{
    if (condition)
    {
        // code
    }
    else
    {
        // code
    }
}

class my_class
{
public:
    my_class();
};
```

### 3.2 缩进与空格

- 使用 **4 个空格** 缩进，**禁止使用制表符（TAB）**。
- 二元操作符左右各留一个空格（`a + b`），逗号后留一个空格。
- 不在行尾留有空白字符。

### 3.3 行宽

- 每行最多 **120** 字符。
- 过长参数列表或长表达式应适当换行，换行后对齐参数：

```cpp
void long_function_name(int parameter_one,
                        int parameter_two,
                        const QString &parameter_three);
```

***

## 4. 命名约定

### 4.1 通用规则

- 所有命名使用**小写字母 + 下划线**风格（Unix / C 传统），**不使用驼峰**。
- 命名应清晰表达语义，避免缩写（公认缩写除外，如 `ip`, `uuid`, `tcp`）。

### 4.2 类/结构体

- 类名、结构体名：**全小写下划线**。

```cpp
class discovery_service { };
class file_receiver { };
struct device_info { };
```

> 说明：此风格为满足用户要求，与传统 Qt 大写驼峰不同，整个项目必须统一。

### 4.3 函数与成员方法

- **全小写下划线**。

```cpp
void start();
bool is_discovery_active();
void set_device_name(const QString &name);
```

- Qt 信号槽也遵循此规则：

```cpp
signals:
    void device_discovered(const device_info &info);
    void transfer_progress(qint64 sent, qint64 total);
```

### 4.4 变量与成员变量

- 局部变量、函数参数、静态变量：**全小写下划线**。
- **类成员变量**：在末尾添加一个下划线 `_` 作为后缀，以区分局部变量。

```cpp
class file_sender
{
private:
    QTcpSocket *socket_;
    QString file_path_;
    qint64 file_size_;
};
```

### 4.5 常量与枚举

- 常量使用 `const` 或 `constexpr`，全小写下划线。

```cpp
const int default_udp_port = 10262;
const QString app_name = "LanDrop";
```

- 枚举类型使用 `enum class`，枚举值全小写下划线：

```cpp
enum class transfer_state
{
    idle,
    connecting,
    in_progress,
    completed,
    failed
};
```

### 4.6 命名空间

- 命名空间使用全小写下划线。
- 项目所有核心代码放在 `landrop` 命名空间内。

```cpp
namespace landrop
{
    namespace core
    {
        // ...
    }
}
```

### 4.7 宏定义

- 宏名称全部 **大写 + 下划线**。应尽量少用宏，优先使用 `constexpr` 或 `enum class`。

```cpp
#define LOG_LEVEL_DEBUG 0
```

## 5. 注释与文档

### 5.1 文件头注释

- 每个源文件开头使用统一格式的 Doxygen 风格注释：

```cpp
/**
 * @file discovery_service.cpp
 * @brief 局域网设备发现服务
 * @author Mupengyi
 * @date 2026-05-02
 */
```

### 5.2 函数注释

- 公共接口必须注释，描述功能、参数、返回值及可能抛出的异常。

```cpp
/**
 * @brief 开始监听组播消息
 * @param port  UDP 端口号
 * @return 监听成功返回 true，否则 false
 */
bool start(quint16 port);
```

### 5.3 实现注释

- 只在复杂逻辑或非直观代码处添加行内注释。
- 注释使用 `//` 或 `/* */`，风格视团队习惯统一。建议简单注释用 `//`。
- 保留 `TODO`、`FIXME` 标记，并注明负责人。

```cpp
// TODO(zhangsan): 添加 IPv6 支持
```

## 6. 禁止魔法数字

### 6.1 规则

- **任何具有业务或协议含义的数字都不能直接写在代码中**（即魔法数字）。
- 所有这类数字必须定义为 **命名常量**（`const` 或 `constexpr`）、**枚举值**或**宏**（仅在必须用于预编译时使用）。
- 以下示例是**被禁止**的：

```cpp
// 错误示例
if (state == 3) { ... }
socket->bind(QHostAddress::Any, 10262);
timer->start(30000);
```

- 正确做法：

```cpp
// 正确示例
constexpr int transfer_state_in_progress = 3;
if (state == transfer_state_in_progress) { ... }

constexpr quint16 udp_discovery_port = 10262;
socket->bind(QHostAddress::Any, udp_discovery_port);

constexpr int broadcast_interval_ms = 30000;
timer->start(broadcast_interval_ms);
```

### 6.2 适用范围

- 端口号、缓冲区大小、超时时间、重试次数、协议版本号、文件块大小、魔数前缀等。
- 纯程序逻辑上的 0、1、-1 等不具业务含义的数字可直接使用（如 `index = 0`、`length - 1`、`return -1` 表示错误），但鼓励在可能混淆的场景下定义常量。

***

## 7. Qt 特定约定

### 7.1 对象所有权

- 依赖 Qt 的父子内存管理，界面控件必须指定父对象，业务对象（无父子关系）由智能指针或显式 `delete` 管理。
- 推荐使用 `QScopedPointer` 或 `std::unique_ptr` 管理非 QObject 对象的生命周期。

### 7.2 信号槽连接

- 使用新型信号槽语法（函数指针形式），禁止使用 `SIGNAL()` / `SLOT()` 宏。

```cpp
connect(sender, &sender_class::signal_name,
        receiver, &receiver_class::slot_name);
```

### 7.3 UI 与逻辑分离

- `.ui` 文件仅包含界面布局，不添加业务代码。
- 界面类只负责展示与交互，通过信号调用核心类方法，不得直接编写网络或文件逻辑。

***

## 8. 其他规则

### 8.1 使用 `nullptr`

- 一律使用 `nullptr`，禁止使用 `NULL` 或 `0` 来表示空指针。

### 8.2 使用 `auto`

- 在冗长类型名或迭代器声明时谨慎使用 `auto`，但仍须保证可读性。简单类型（`int`, `double` 等）勿用 `auto`。

### 8.3 `#include` 顺序

- 顺序：本类的头文件 → 项目内其他头文件 → Qt 库头文件 → 标准库头文件。
- 每组之间空一行。

```cpp
#include "discovery_service.h"

#include "protocol.h"

#include <QUdpSocket>
#include <QJsonObject>

#include <memory>
```

### 8.4 预编译头

- 不强制使用预编译头，但在大型项目中可适当添加以加快编译速度。
