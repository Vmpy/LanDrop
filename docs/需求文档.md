根据搜索结果与分析，端口 `10262` **未被 IANA 分配给已知服务，但历史上曾有个别恶意软件使用过该端口**。考虑到 `LanDrop` 仅限于局域网内部使用，这一历史情况基本不会带来影响，你完全可以将其用作组播端口。

以下是为 `LanDrop` 更新的完整版需求规格说明书，已应用新端口及关键配置项。

## 1. 引言

### 1.1 项目背景

在同一个 WiFi 热点或有线局域网环境下，经常需要将文件从一台电脑传到另一台，或跨平台共享。传统手段如 U 盘、IM 工具或共享文件夹不够直接。LanDrop 旨在提供一个轻量、无需互联网、无需账号、即开即用的局域网文件传输工具，仅供内部调试和轻量使用场景。

### 1.2 目标用户

- 公司内部开发、测试人员
- 同一物理空间内需要快速交换文档的协作人员
- 临时传递大文件的团队（设计稿、固件包、日志包等）

### 1.3 项目范围

- **包含**：设备发现、用户选择设备、文件选择与发送、接收确认、传输进度、错误处理。
- **不包含**：互联网传输、NAT 穿透、用户认证与账号系统、端到端加密、文件夹同步、磁盘映射等。

***

## 2. 总体描述

### 2.1 产品视角

LanDrop 是一个**桌面 GUI 应用程序**，运行时在系统托盘常驻，打开主窗口显示局域网内其他运行中的 LanDrop 设备列表。用户可以拖拽文件或点击按钮选择文件，发送给选中的设备。接收方收到通知后选择保存位置，即可完成传输。

### 2.2 运行环境

- **操作系统**：Windows 10 及以上，macOS 11+，Linux (主流发行版)
- **Qt 版本**：Qt 6.5 LTS 及以上，优先使用 Qt 6.5 系列稳定版
- **网络**：IPv4/IPv6 局域网，设备间可互相 Ping 通
- **权限**：监听本地端口、访问文件系统（读取/写入）
- **编译器**：MSVC 2022 / GCC 11+ / Clang 14+

### 2.3 设计约束

- 所有通信仅发生在局域网内，**不经过任何外部服务器**
- 使用 **LGPLv3 动态链接** Qt 库，确保合规分发
- 避免使用 Qt 的 GPL 模块（如 Qt Charts 商业部分等）
- 对外部依赖尽量少，只用 Qt 内置网络模块
- 使用C++代码动态加载UI

## 3. 功能需求 (Functional Requirements)

### 3.1 设备发现

**FR-01. 设备在线广播**

- 应用程序启动后，后台通过 **UDP 组播**（地址 239.255.255.250，端口 **10262**）周期发送“在线声明”。
- 广播报文格式为 JSON，包含：
  - `version`：协议版本（如 1）
  - `device_name`：用户自定义的设备显示名（默认取计算机名）
  - `uuid`：设备唯一标识（首次生成并持久化存储）
  - `tcp_port`：文件接收监听端口（动态分配，随后通过广播公布）
  - `platform`：操作系统标识（win/mac/linux）
- 发送间隔：启动时立即发送，之后每 30 秒发送一次，关闭时发送“离线”报文。

**FR-02. 设备列表维护**

- 应用程序监听同一组播地址，接收其他设备的在线报文。
- 根据 UUID 维护本地设备列表，记录其 IP 地址、设备名、端口等。
- 若超过 90 秒未收到某个设备的再次广播，则从列表移除该设备（视为离线）。
- 收到新的广播报文时更新对应设备信息。

### 3.2 连接管理

**FR-03. 建立 TCP 连接**

- 发送方选中目标设备后，使用对方的 `tcp_port` 和 IP 启动一个 `QTcpSocket` 主动连接。
- 连接成功后，发送方先发送握手包：
  ```json
  {
    "type": "handshake",
    "device_name": "发送方名称",
    "uuid": "发送方UUID"
  }
  ```
- 接收方验证后返回确认包 `{"type": "handshake_ack"}`，否则断开。

**FR-04. 连接维持与心跳**

- 连接建立后，每隔 15 秒互发心跳报文 `{"type":"ping"}` / `{"type":"pong"}`。
- 超过 45 秒未收到任何数据则判定超时，主动断开并通知用户。

### 3.3 文件发送与接收

**FR-05. 文件选择**

- 发送方通过以下方式添加待发送文件：
  - 点击 “添加文件” 按钮打开系统文件对话框
  - 拖拽文件或文件夹至主窗口（v1.0 先支持单文件传输）
- 在发送列表中显示文件名、大小、状态。

**FR-06. 传输协议数据包**

- 文件数据通过 **二进制分块传输**，格式如下：
  - **文件元数据包**（JSON 文本，后跟 `\n` 结尾）：
    ```json
    {
      "type": "file_meta",
      "file_name": "example.zip",
      "file_size": 1048576,
      "chunk_size": 8192
    }
    ```
  - **文件数据块包**（二进制）：
    - 4 字节块序列号（大端）
    - 4 字节有效载荷长度（大端）
    - 载荷数据（实际文件内容片段）
    - 4 字节 CRC32 校验和（载荷数据）
  - **确认包**：接收方每收到 64 块或 1 秒间隔返回 ACK，包含最后连续接收的块号，发送方据此进行选择性重传或继续发送。

**FR-07. 接收确认与保存**

- 接收方弹出提示（即使窗口最小化也要通知）：“`发送方名称` 想发送 `文件名 (大小)`，是否接收？”。
- 用户可选：“接收并保存”、“拒绝”。
- 若接受，弹出另存为对话框选择保存路径。
- 接收过程中实时显示进度条、速度、剩余时间估算。

### 3.4 传输进度与后台发送

**FR-08. 进度展示**

- 主窗口实时更新整体队列进度。
- 单个文件传输显示已传输字节/总字节、进度百分比、当前速度（MB/s）。
- 速度采用滑动窗口平滑算法计算。

**FR-09. 多文件传输**

- 发送方支持添加多个文件形成队列，依次发送。
- 接收方每次只处理一个连接，多个发送方排队等候（未来可并发）。

**FR-10. 传输完成与通知**

- 传输成功后，双方显示 “传输完成” 通知，发送方移除文件项，接收方打开文件所在文件夹按钮可选。
- 若传输中断（网络断连、取消等），提示失败信息，已传输的部分临时文件自动删除。

### 3.5 用户设置

**FR-11. 设备显示名**

- 允许用户在设置中修改设备显示名，修改后需重新广播。
- 保存设备名和 UUID 至 `QSettings` 或本地 JSON 配置。

**FR-12. 下载路径配置**

- 默认保存路径为用户`下载`目录，可在设置中更改。

## 4. 非功能需求 (Non‑Functional Requirements)

### 4.1 性能

- 局域网千兆环境，单文件传输速度应接近网络带宽的 90%。
- 设备发现延迟 ≤ 5 秒。
- 内存占用不超过 80 MB（空载），传输时平滑增长，不出现界面卡顿。

### 4.2 可用性

- 界面简洁，无菜单栏，核心操作不超过两次点击。
- 所有异步操作均通过信号槽更新 UI，响应迅速。
- 支持系统托盘最小化到托盘运行，有新传输请求时气泡通知。

### 4.3 可靠性

- TCP 连接异常断开时，自动清理资源，不残留端口占用。
- 接收到的文件校验失败时，丢弃错误块并请求重传，重传超过 3 次则终止并报错。

### 4.4 兼容性

- 可运行在 Windows 10/11、macOS 11+、Ubuntu 22.04 LTS。
- 确保 IPv4 和 IPv6 双栈环境正常。

### 4.5 安全性

- 仅响应同一子网内的设备请求（或组播 TTL=1）。
- 接收文件前用户确认，防止未经授权的文件写入。
- 不执行接收到的任何可执行代码，不自动运行文件。

### 4.6 可维护性

- 核心协议与 UI 分离，可单独测试。
- 日志记录到文件，方便调试（可配置日志级别）。

## 6. 架构概要

### 6.1 整体架构

采用 **模型-视图-控制器 (MVC)** 分层，Qt 信号槽驱动：

- **Core 模块（无 GUI）**
  - `DiscoveryService`：封装 UDP 组播收发，维护设备表
  - `TransferServer`：TCP 服务端，处理连接接受、握手、接收文件
  - `TransferClient`：单个 TCP 客户连接，处理发送队列、分块、重传
  - `Protocol`：提供包序列化/反序列化、校验
- **UI 模块（Widgets）**
  - `MainWindow`、`DeviceListWidget`、`SendPanel`、`ReceiveDialog`
  - 通过信号连接Core模块，UI 无业务逻辑

### 6.2 关键类设计（简要）

- `DiscoveryService(QObject)`
  - `void start(quint16 udpPort)`
  - signals: `deviceDiscovered(DeviceInfo)`, `deviceOffline(QString uuid)`
- `TransferServer(QObject)`
  - 接受连接后创建 `ReceiverHandler`
- `FileReceiver(QObject)`
  - signals: `fileProposal(SenderInfo, FileMeta)`, `progress(qint64, qint64)`, `finished()`
- `FileSender(QObject)`
  - slots: `sendFile(QString filePath)`
  - signals: `progress(qint64, qint64)`, `finished()`

## 7. 协议规范 v1.1 (详细)

### 7.1 发现协议 (UDP 组播)

- 地址：`239.255.255.250:10262`
- 报文：
  ```json
  {
    "type": "announce",
    "version": 1,
    "device_name": "MyPC",
    "uuid": "550e8400-e29b-...",
    "tcp_port": 54322,
    "platform": "win"
  }
  ```
- 离线报文：`type: "goodbye"`，携带相同 uuid。

### 7.2 控制通道协议 (TCP JSON)

- 握手：`{"type":"handshake","device_name":"...","uuid":"..."}`
- 心跳：`{"type":"ping"}` / `{"type":"pong"}`
- 拒绝：`{"type":"reject","reason":"User declined"}`
- 开始发送：`{"type":"send_start","files":[{"name":"...","size":...}]}` （v1.1 多文件预告）

### 7.3 二进制数据块格式

| 字段      | 字节数 | 说明          |
| ------- | --- | ----------- |
| seq     | 4   | 块序号（从0开始）   |
| len     | 4   | 有效载荷长度 (N)  |
| payload | N   | 文件数据        |
| crc32   | 4   | 载荷 CRC32 校验 |

### 7.4 错误与重传

- 接收方按 `seq` 组装文件，如发现缺失块或 CRC 错误，发送 `{"type":"nack","seq":5}` 请求重传指定块。
- 发送方维护未确认块列表，收到 nack 或超时后重发。
